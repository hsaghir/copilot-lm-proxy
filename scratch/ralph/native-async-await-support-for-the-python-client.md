# Native Async/Await Support for the Python Client

The copilot_proxy Python client is entirely synchronous, which is a fundamental architectural mismatch with the modern Python AI ecosystem. Every major AI agent framework (LangGraph, LangChain, AutoGen, CrewAI), every async web framework (FastAPI, Starlette, aiohttp), and every async notebook environment (JupyterLab) expects async-compatible APIs. Using the current synchronous client inside any of these contexts blocks the event loop, degrades throughput, and makes parallel model calls impossible without resorting to threads.

The fix is to add an AsyncCopilotClient class alongside the existing CopilotClient, plus module-level async_ask, async_chat, and async_list_models convenience functions. Python 3.10+ has asyncio built-in, so this can be done with zero new runtime dependencies — preserving the project's core 'zero dependencies' philosophy. Non-blocking HTTP can be implemented via asyncio.to_thread() wrapping the sync urllib calls, which is idiomatic, robust, and avoids reimplementing raw HTTP over asyncio streams. Async streaming uses asyncio.Queue to bridge the sync SSE reader thread with an async generator consumer.

This single structural change unlocks the proxy for the entire class of async Python applications: users can await model calls in FastAPI route handlers without blocking other requests, fire off parallel queries to multiple models and collect results with asyncio.gather(), consume streamed responses with async for in websocket handlers, and compose the client naturally inside async agent orchestration loops. It doubles the surface area of the public API in a fully backward-compatible way and positions the library as a first-class citizen in modern Python AI development.
